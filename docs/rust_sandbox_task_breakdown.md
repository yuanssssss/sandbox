# Rust 沙箱开发任务拆分表

## 1. 说明

本文基于前面的沙箱设计文档，给出一个适合实际开发推进的任务拆分表。  
目标是让项目能够按阶段落地，而不是一次性把所有安全特性堆上去。

推荐原则：

- 先做可运行 MVP
- 再做隔离
- 再做资源限制
- 再做防御加固
- 最后做工程化和回归测试

状态说明：

- `[x]` 已完成
- `[-]` 部分完成
- `[ ]` 未开始

| [x] | T002 | 建立统一错误类型 | 统一内部错误边界 |
| [x] | T003 | 建立日志框架 | 便于后续调试复杂失败路径 |
| [x] | T004 | 定义任务配置模型 | 把限制和挂载策略模型化 |

## 6. 推荐 issue 划分方式

如果你要把这些任务直接放到 issue 系统里，建议按以下粒度拆分：

### Epic

- Epic 1：最小执行器
- Epic 2：文件系统与 namespace 隔离
- Epic 3：cgroup 资源限制
- Epic 4：权限和 seccomp 强化
- Epic 5：测试与上线准备

### 单个 issue 粒度建议

一个 issue 最好满足：

- 1 到 2 天内能完成
- 有明确输入和输出
- 有可以自动化验收的结果

例如：

- “实现 `memory.max` 写入和结果读取”
- “增加 network namespace 并验证无法访问外网”
- “支持将 stdout/stderr 重定向到任务目录”

## 7. 推荐实现顺序

如果资源有限，建议按这个顺序推进：

1. T001-T013  
   先做最小可执行 MVP

2. T014-T023  
   再做 namespace 与 rootfs 隔离

3. T024-T030  
   加入 cgroup v2 资源限制

4. T031-T038  
   再做用户隔离、能力裁剪和 seccomp

5. T039-T047  
   最后补审计、回归和上线前验证

## 8. 最小可用版本定义

如果你想先做一个能演示、能迭代的版本，MVP 建议包含：

- CLI 执行入口
- 输出重定向
- wall-clock 超时
- mount namespace
- PID namespace
- network namespace
- 最小 rootfs
- cgroup v2 的 `memory.max` 和 `pids.max`

这个版本先不强求：

- seccomp
- 多 profile
- 复杂服务端编排
- 完整审计平台

但必须保证：

- 能正确清理
- 不泄漏宿主机文件系统
- 不允许联网
- 不允许无限创建进程

## 9. 上线前检查清单

上线前至少确认：

- 所有失败路径都能清理 cgroup 和挂载点
- 超时后整棵进程树会被杀死
- `/proc` 不泄漏敏感信息
- 可写目录默认 `nodev,nosuid,noexec`
- 默认 profile 下常见语言可运行
- 高风险 syscall 已被覆盖或被 namespace 限制
- 恶意样例回归全部通过

建议以 `docs/security_review_checklist.md` 作为正式发布检查单，而不是仅依赖本节摘要。

## 10. 结论

这个任务拆分的核心思路是：

- 先保证“能跑”
- 再保证“跑在边界里”
- 再保证“跑不炸机”
- 最后保证“出了问题也能定位和恢复”

如果按这个顺序推进，项目会更稳，也更容易在中途验证设计是否真的成立。
